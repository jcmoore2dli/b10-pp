  requeueSubmission    — Admin callable, clears doc, reruns pipeline
  requestPass2         — Instructor/admin callable, calls runPass2()

  processSubmission() — 14-step shared pipeline:
    Step 1:  Counter transaction → server-side submissionNumber assignment
    Step 2:  Status → "processing"
    Step 3:  Audio fetch from Firebase Storage
    Step 4+5: Deepgram STT → raw transcript + words[] (disfluency preserved)
    Step 6:  Disfluency pre-processor (LEVEL2 tasks only)
    Step 7+8: EVALUATION_POLICY injection + Claude prompt routing by taskType
    Step 9+10: Claude scoring → JSON parse + validate
    Step 11: pass2Available flag logic
    Step 12+13: All result fields written → status "complete"
    Step 14: Error → status "failed"
  Retry wrapper: 3 attempts, 2s/4s/8s backoff on Deepgram + Claude

  runPass2() — AssemblyAI clean transcript for instructor review only
    Score 1 gate enforced (hard block, defense in depth)
    Output written ONLY to transcriptClean — NEVER reaches scoring engine

SECTION 3 — DO NOT REOPEN THESE DECISIONS
════════════════════════════════════════════════════════════════════════

These are locked. Do not revisit without explicit instruction from J. Moore.

ILR target values: "2" and "3" only. No "2+" anywhere in schema.
Score field ESO: integer 1–4. LEVEL2: integer 1–3. Not ILR levels.
ILR labels in student feedback: Out of scope. Never.
corpusType "CC": It is "COR". Do not reintroduce "CC".
Track on /students: immutable after creation. trackOverrideNote required for override.
PSU as scaffold axis: Not selectable. Always implicit. argument_structure foregrounds it.
MF vocab in scaffold system: Not included. Phase 4 evaluation layer only.
Auto-generated note summary: On-demand only. Never on note write.
EXT as umbrella: EXT is a subtype. One canonized long-form family.
STT Pass 1: Deepgram (disfluency preserved). NOT Whisper. NOT AssemblyAI.
STT Pass 2: AssemblyAI (conditional, instructor-facing only). NOT scoring engine input.
Scoring engine: Claude Sonnet, temperature = 0. Non-negotiable.
Raw transcript: NEVER smoothed before reaching scoring engine.
submissionNumber: Server-side transaction only. Client always writes 0.
Score 1 + Pass 2: Permanently incompatible. Hard block in code.

Auth method: Email/password only. Anonymous and phone auth ruled out.
Entry flow (Option C): Access code only on EntryScreen. b10Id always
  comes from custom claims — never from user input. Student never types
  their own ID. Access code validation deferred to Week 2 enrollment flow.
b10Id is a backend pseudonym: Students never see or type it. Instructors
  use it as a lookup key. Like a medical record number — internal only.
Instructor student lookup: Two paths — (1) by name/email for discovery,
  (2) by b10Id for fast instructor-to-instructor hand-off (e.g. SLT teacher).
  Both paths return the same student record view. Build in Week 4 dashboard.
Role authority: Custom claims are the single source of truth for role.
  The /users collection is NOT used for role checks in Firestore rules.

SECTION 3B — KNOWN ISSUES AND TECHNICAL DEBT
════════════════════════════════════════════════════════════════════════

These are not blockers but must be addressed before or during pilot.

ISSUE-1 (HIGH): AuthContext Vite Fast Refresh incompatibility
  During heavy file editing sessions, Vite HMR invalidates the AuthContext
  component tree. Symptom: buttons stop responding with no console errors.
  Root cause: useAuth() hook and AuthProvider exported from same file.
  Fix: Split into separate files —
    AuthContext.js   — context object only
    AuthProvider.jsx — provider component
    useAuth.js       — hook only
  This fully resolves HMR fragility. Schedule: early Week 2.
  Workaround: full dev server restart + hard browser refresh (Ctrl+Shift+R)

ISSUE-2 (MEDIUM): requeueSubmission + requestPass2 role checks use /users
  These two Cloud Functions check role by reading from /users collection,
  which does not exist in the current architecture. They should check
  request.auth.token.role (custom claims) instead.
  Fix: Update both functions in functions/index.js before pilot.
  Schedule: Week 2 or Week 3.

ISSUE-3 (LOW): entered state resets on every sign-in
  AppInner uses React state (entered: bool) to gate /passages route.
  This resets to false on every sign-in, always showing EntryScreen first.
  This is CORRECT behavior for now but is held together by state +
  sessionStorage rather than a single source of truth.
  Fix: Clean up when enrollment flow is built in Week 2.

ISSUE-4 (LOW): Dev URL must include trailing slash
  Always use: http://localhost:5173/b10_practice_platform/
  Without trailing slash, router fails to match routes.
