## DESIGN PHILOSOPHY

Build for validity first. Optimize for cost later.

This architecture prioritizes scoring construct validity over infrastructure
efficiency. Cost optimization decisions should be deferred until real usage data
is available (target: September/October 2026 active deployment).

CORE VALIDITY PRINCIPLE — ARCHITECTURAL INVARIANT:
The scoring engine receives the raw Deepgram transcript only, on every response,
without exception. This invariant must be enforced in ClaudeScorer.js and verified
before deployment. No smoothed or cleaned transcript ever reaches the scoring engine
under any condition.

CONSTRUCT PRINCIPLE:
ESO tasks (Level 3 target):
  Score 2 = Level 3 not functionally present (ILR 2)
  Score 3 = Level 3 functionally present but inconsistent (ILR 2+)
  Score 4 = Level 3 sustained and controlled (ILR 3)
  Primary scoring validity risk zone: 2/3 boundary (reasoning gate).

Narration tasks (Level 2 target):
  Score 1 = Level 2 not functionally realized (ILR 1)
  Score 2 = Level 2 developing (ILR 1+)
  Score 3 = Level 2 sustained (ILR 2)
  Primary scoring validity risk zone: 1/2 boundary (past tense gate).

Description tasks (Level 2 target):
  Score 1 = Level 2 not functionally realized (ILR 1)
  Score 2 = Level 2 developing (ILR 1+)
  Score 3 = Level 2 sustained (ILR 2)
  Primary scoring validity risk zone: 1/2 boundary (spatial reference gate).

Instructions tasks (Level 2 target):
  Score 1 = Level 2 not functionally realized (ILR 1)
  Score 2 = Level 2 developing (ILR 1+)
  Score 3 = Level 2 sustained (ILR 2)
  Primary scoring validity risk zone: 1/2 boundary (sequential logic gate).

Disfluency is criterion-level evidence at all task type boundaries.

## ARCHITECTURE REVISION NOTE (v1.1 → v1.2)

v1.1 specified dual-pass STT (Deepgram + AssemblyAI) as default processing for all
responses. v1.2 revised this based on norming evidence:

Finding 1: In the majority of responses, disfluency does not interfere with scoring.
Scores are determined primarily by argument structure, causal reasoning, and sustained
control — not transcription clarity.

Finding 2: Criterion-relevant disfluency (false starts revealing inability to complete
complex structures) occurs in a minority subset of responses. It is the exception,
not the rule.

Finding 3: Default dual-pass creates unnecessary architectural complexity and a
standing risk that a future configuration change could inadvertently route a smoothed
transcript to the scoring engine — a systematic validity threat.

Decision: AssemblyAI Pass 2 moved from default to conditional/optional.
The simplified default pipeline is the correct architecture for this construct.

## STT LAYER — DETAILED SPECIFICATION

### Default: Deepgram (All Responses — Scoring Input)

Purpose:
  Produce a raw transcript preserving all disfluency signals relevant to
  criterion-level scoring. This is the SOLE input to the scoring engine
  on every response without exception.

Configuration:
  - Disfluency preservation: ENABLED
  - Filler word retention (uh, um, eh): ENABLED
  - False start retention: ENABLED
  - Smart formatting: DISABLED
  - Punctuation: ENABLED
  - Speaker diarization: ENABLED (isolate student speaker before scoring)

Why raw transcript fidelity matters:
  The rubric at each task type distinguishes emergent from sustained control.
  A student who initiates a complex structure and cannot complete it produces
  a false start — evidence of emergent control, not sustained control. If the
  STT smooths the false start into a completed structure, the scoring engine
  sees surface competence that does not exist in the student's production.

  This pattern occurs in a minority subset of responses but is diagnostically
  significant when it does occur — particularly at gate boundaries. The
  default pipeline preserves this evidence on every response at minimal cost,
  ensuring it is never lost for the cases where it matters.

Free tier: Deepgram provides 12,000 free minutes/month. Anticipated B10-PP
usage volumes are expected to remain within this tier through initial deployment.

### Conditional: AssemblyAI (Triggered Only — Not Default)

Purpose:
  Produce a clean, readable transcript for instructor review or student
  conference when the raw transcript is insufficient for that purpose.
  AssemblyAI output is NEVER routed to the scoring engine under any condition.

Trigger conditions:
  TRIGGER 1 — Non-empty transcript_note:
    If the scoring engine returns a non-empty transcript_note field,
    criterion-relevant disfluency has been flagged. Instructor-initiated
    Pass 2 becomes available for a clean reference transcript to support
    review and student conference.

  TRIGGER 2 — Instructor manual request:
    Instructor may request a clean transcript for any response via the
    instructor dashboard. Pass 2 runs on demand.

  SCORE 1 GATE:
    Pass 2 is never triggered on Score 1 responses regardless of other
    conditions. A Score 1 response is too brief, disfluent, or off-topic
    for a clean transcript to add meaningful value.

Configuration:
  - Smart formatting: ENABLED
  - Filler word removal: ENABLED
  - Punctuation and casing: ENABLED
  - Speaker diarization: ENABLED

Free tier: AssemblyAI provides comparable free tier. Conditional use further
reduces expected call volume below default-pass levels.

### STT Validity Demonstration Case

Recorded April 2026. High-disfluency profile. Topic: minimum wage regulation.

This case illustrates why raw transcript fidelity matters even when it is
not the majority case — and why the transcript_note trigger is the correct
mechanism for identifying responses where Pass 2 adds value.

RAW (Deepgram — what scoring engine receives):

  "So in my opinion, it depends, uh, on the country and especially on the
  culture of that country. Because, uh, doing the example of Italy, we already
  had, uh, we already had the time, uh, in which the... there was a minimum
  degree, uh, a minimum salary, sorry... for, uh, that was really high for
  almost, uh, all the people that were working. And it created a huge economical
  crisis, uh, because, uh, it involved people, uh, to, uh, basically don't work.
  So, uh, 'cause of this, uh, I would say that, uh, even though many people
  would consider the idea of introducing a, a minimum, uh, salary or increasing
  it were, were present, uh, I would say that is not a, a good idea... because,
  uh, uh, the problem that it creates to Italy, the main problem that it created,
  uh, to, to our government was, uh, the scarcity of funds, uh, into the... mm,
  more... eh, into the government's, um (snaps fingers) scarcity of funds, uh,
  for the governments basically, for all the other expenses."

Scoring engine verdict: Score 2 — Partial
transcript_note output: "False starts at two critical junctures reveal inability
to complete initiated complex structures: concessive collapses ('were, were
present') and institutional framing abandoned twice ('scarcity of funds, uh,
into the...'). These were factored into the grammatical complexity assessment."

Result: Non-empty transcript_note → Pass 2 available on instructor request.

SMOOTHED (AssemblyAI — what must NEVER reach scoring engine):

  "In my opinion, it depends on the country and especially on the culture of
  that country. Because doing the example of Italy, we already had a time in
  which there was a minimum salary that was really high for almost all the
  people that were working, and it created a huge economic crisis because it
  involved people to basically not work. So because of this, I would say that
  even though many people would consider the idea of introducing or increasing
  a minimum salary, I would say that it is not a good idea, because the main
  problem that it created to our government was the scarcity of funds for all
  the other expenses."

Scoring engine verdict on smoothed: borderline Score 2/3.
Concessive appears to complete. Institutional framing appears to resolve.
Emergent control evidence erased. Score inflated.

CONCLUSION: The transcript_note trigger correctly identifies this response
as a candidate for Pass 2. The invariant correctly prevents the smoothed
transcript from reaching the scorer.

## SCORING ENGINE — DETAILED SPECIFICATION

Engine: Anthropic Claude — claude-sonnet-4-20250514

System prompts — routed by taskType:
  taskType === 'ESO'          → Section 8 Prompt 4 ESO v1.5 Rev.4
  taskType === 'NARRATION'    → Section 8 Prompt 4b Past Tense Narration v1.2
  taskType === 'DESCRIPTION'  → Section 8 Prompt 4c Description v1.1
  taskType === 'INSTRUCTIONS' → Section 8 Prompt 4d Instructions v1.4
  Deploy full prompt text as system parameter in API call.
  Do not prepend to user message.

Temperature: 0
  Scoring must be deterministic. Erratic scores on identical inputs =
  temperature not set correctly. Verify in ClaudeScorer.js before deployment.

Input:
  PROMPT DESCRIPTION:
  {promptDescription}

  SCAFFOLD FOCUS:
  ESO tasks:
    focusMode: {focusMode}
    primaryTarget: {primaryTarget}
    secondaryTarget: {secondaryTarget}
  Narration, Description, Instructions tasks:
    primaryFocus: {primaryFocus}
    activeMonitors: {activeMonitors}

  STUDENT TRANSCRIPT:
  {transcript}

  Where {transcript} = Deepgram raw transcript, student speaker only.
  Speaker diarization must be applied before passing to scorer.

  DISFLUENCY METADATA (Narration, Description, Instructions — passed alongside
  transcript for silent pause and pacing signals):
  - Silent pauses >1.5s: {pause_count_1500} (at {pause_timestamps_1500})
  - Silent pauses >2.5s: {pause_count_2500} (at {pause_timestamps_2500})
  - Mean inter-word gap: {mean_gap}s
  - Filled pauses (uh/um/eh): {filled_pause_count}

  NOTE: Disfluency metadata block omitted for ESO tasks. ESO scoring relies
  on transcript-visible disfluency signals only (false starts, structural
  collapses). Metadata block active for all ILR 2 task types.

Output: Valid JSON

  ESO tasks:
  {
    "score": <1, 2, 3, or 4>,
    "score_label": "<Insufficient|Partial|Good|Excellent>",
    "strengths": "<string>",
    "gaps": "<string>",
    "scaffold_feedback": "<string — omit if Holistic mode>",
    "language_feedback": "<string>",
    "transcript_note": "<string — empty if none>"
  }

  Narration tasks:
  {
    "score": <1, 2, or 3>,
    "score_label": "<Not functionally realized|Developing|Proficient>",
    "strengths": "<string>",
    "gaps": "<string>",
    "language_feedback": "<string>",
    "monitor_notes": "<string — empty if no monitors active>",
    "transcript_note": "<string — empty if none>"
  }

  Description tasks:
  {
    "score": <1, 2, or 3>,
    "score_label": "<Not yet|Developing|Proficient>",
    "strengths": "<string>",
    "gaps": "<string>",
    "language_feedback": "<string>",
    "monitor_notes": "<string — empty if no monitors active>",
    "transcript_note": "<string — empty if none>"
  }

  Instructions tasks:
  {
    "score": <1, 2, or 3>,
    "score_label": "<Not yet|Developing|Proficient>",
    "strengths": "<string>",
    "gaps": "<string>",
    "language_feedback": "<string>",
    "monitor_notes": "<string — empty if no monitors active>",
    "transcript_note": "<string — empty if none>"
  }

transcript_note field handling:
  Non-empty → flag in instructor dashboard; make Pass 2 available on request.
  Empty → no action; Pass 2 not triggered.

Alternate engine: OpenAI via ScoringService.js — one-line swap, no frontend impact.

  Raw Deepgram transcript
  Pass 2 clean transcript — available on request if transcript_note non-empty,

## SCORING SERVICE — IMPLEMENTATION NOTES

ClaudeScorer.js requirements:
  1. Temperature = 0 — verify before deployment
  2. Route to correct system prompt based on taskType:
     taskType === 'ESO'          → Prompt 4 ESO v1.5 Rev.4
     taskType === 'NARRATION'    → Prompt 4b Narration v1.2
     taskType === 'DESCRIPTION'  → Prompt 4c Description v1.1
     taskType === 'INSTRUCTIONS' → Prompt 4d Instructions v1.4
  3. Raw Deepgram transcript only as {transcript} — invariant, no exceptions
  4. Speaker diarization applied before passing to scorer
  5. ESO tasks: JSON response validated — score must be integer 1–4
  6. Narration, Description, Instructions tasks: JSON response validated —
     score must be integer 1–3
  7. score_label must match score per task-family mapping:
     ESO:          1=Insufficient · 2=Partial · 3=Good · 4=Excellent
     Narration:    1=Not functionally realized · 2=Developing · 3=Proficient
     Description:  1=Not yet · 2=Developing · 3=Proficient
     Instructions: 1=Not yet · 2=Developing · 3=Proficient
  8. transcript_note parsed — non-empty value triggers Pass 2 availability flag
  9. taskFamily field passed to frontend:
     ESO          → 'LEVEL3'
     Narration    → 'LEVEL2'
     Description  → 'LEVEL2'
     Instructions → 'LEVEL2'
     (used by UI for score display denominator and contextual line)
  10. Disfluency metadata pre-processor active for all LEVEL2 task types.
      Compute from Deepgram word-level timestamps before passing to scorer:
      - Silent pause counts >1.5s and >2.5s with timestamps
      - Mean inter-word gap
      - Filled pause count (uh/um/eh tokens)
      ESO tasks: metadata block omitted.

ScoringService.js routing:
  taskType === 'ESO'          → ClaudeScorer.js with Prompt 4 ESO
  taskType === 'NARRATION'    → ClaudeScorer.js with Prompt 4b
  taskType === 'DESCRIPTION'  → ClaudeScorer.js with Prompt 4c
  taskType === 'INSTRUCTIONS' → ClaudeScorer.js with Prompt 4d
  engine === 'openai'         → OpenAIScorer.js (alternate, any task type)

Pass 2 trigger logic:
  if (score === 1) → Pass 2 never triggered
  if (transcript_note !== '') → flag Pass 2 available; trigger on instructor request
  if (instructor_manual_request) → trigger Pass 2 regardless of transcript_note
  default → Pass 2 not triggered

## COST ARCHITECTURE

Built for validity. Optimize after September/October 2026 deployment data.

Current cost profile:
  Deepgram (every response):    expected within free tier
  AssemblyAI (conditional):     triggered subset only — low volume expected
  Claude API (every response):  dominant cost ~$0.015–0.020/call

The simplified architecture reduces AssemblyAI calls without reducing scoring
validity. The cost reduction is a secondary benefit of the correct architectural
decision, not the reason for it.

Cost optimization levers — DO NOT ACTIVATE pre-deployment data:
  Score 1 gate (implemented) — Pass 2 never triggered on Score 1
  Session batching — off-peak API calls
  Caching — identical transcript/prompt pairs
  Model evaluation — assess Haiku for scoring (requires norming validation first)
