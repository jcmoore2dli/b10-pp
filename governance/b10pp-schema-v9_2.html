Four Focus Modes
holistic
Default. Claude evaluates all ESOP dimensions equally. No scaffold shown to student. PSU always implicitly evaluated.
argument_structure
Claude foregrounds PSU evaluation: point → support → analysis arc. Plain cue shown to student. No additional selection needed.
discourse_frame
Claude gives primary attention to selected frame. Requires primaryFrame. Holistic evaluation continues alongside.
grammar_structure
Claude gives primary attention to selected structure. Requires primaryStructure. Holistic evaluation continues alongside.
combined
Claude attends to both selected frame and structure. Requires both. Student sees both plain cues.
SLT Instructor Access — No Restrictions
SLT instructors have full library access. assignmentType = "slt" is a label only — it does not restrict what they can assign. The supplemental pool (poolType = "supplemental") is additive — more options, not fewer. Students receiving only SLT need unrestricted Track A access. Text-prompt tasks (ESOP, hypothesizing, narration, description, instructions) are generatively infinite — no supplemental pool needed for these.

ILR Target — Canonical Definition

Critical — ilrTarget has exactly two values: "2" and "3". No "2+" anywhere in the schema.
2+ is an OPI rating outcome — not a task category. No passage trains "2+ behavior." Passages train level 2 or level 3 tasks. The 2+ rating emerges from a 3-level interview when the student shows a solid level 2 floor but a ceiling at level 3.

LEVEL3 tasks (ESO) — 4-point output scale (scores 2–4): Score 2 = Level 3 not functionally present (ILR 2). Score 3 = Level 3 functionally present but inconsistent (ILR 2+). Score 4 = Level 3 sustained and controlled (ILR 3). Score 1 is defined in the output format but never triggered — Level 2 is assumed as the floor. Scoring decisions are made entirely on the basis of Level 3 status.

LEVEL2 tasks (Narration, Description, Instructions) — 3-point output scale (scores 1–3): Score 1 = Level 2 not functionally realized (ILR 1). Score 2 = Level 2 developing (ILR 1+). Score 3 = Level 2 sustained (ILR 2). Scoring decisions are made on the basis of Level 2 status.

mfTier scaffolds within level 3 — all tier 0/1/2 passages are ilrTarget = "3". Track A level 2 task assignments are strategic floor prep for the 3-level interview gateway — not remediation. Most Track A students will never receive them.

ORIENT — Architectural Position

Locked — B10 Architecture Canon v3.0
ORIENT is the curriculum entry point — not a tier and not part of the CORE lexical calibration system.

ORIENT passages introduce students to three core ILR-3 discourse patterns using lexically simplified language before full CORE engagement begins. They are brief, structured, and deliberately limited in scope.

Hard constraints (non-negotiable):
· CEFR B0–B1 lexical ceiling — simpler than even CORE Tier 0
· Exactly three patterns only: Trend → Complication → Reassessment · Phenomenon → Explanation → Implication · Example → Reinterpretation → Broader Meaning
· One pattern per passage — no blending
· No MF system · No tier labels · No EXT/PIL logic
· Full ILR-3 discourse structure maintained — only lexical density is simplified

phase = "orientation" · corpusType = "ORI" · mfTier = "n/a"

ORIENT exists to stabilize discourse recognition prior to lexical calibration. It must never be conflated with CORE Tier 0, which operates under a different system entirely.

CORE — Tier Doctrine

Locked — B10 Architecture Canon v3.0
Tier in CORE is defined ONLY by MF (mid-frequency vocabulary) count. Nothing else.

Structural abstraction — complex sentences, discourse markers, counter-intuitive element, grammar features — exists at ALL tiers. Tier is not defined by structure. Structure remains identical across Tier 0–2.

Tier 0 · MF count: 1–2 maximum · No MF placement requirement · Anglo-Saxon vocabulary preferred · Full discourse structure present · Purpose: pattern exposure with minimal lexical strain · phase = "foundation"

Tier 1 · MF count: 3–6 · ≥1 MF item in Sentence 1 · ≥1 MF item in final sentence · phase = "foundation" (Sets 1–4) or "bridge" (Sets 5–7)

Tier 2 · MF count: 6–10 · Full lexical density band · phase = "bridge" (Sets 5–7) or "target" (Sets 8–12)

MF audit status: CORE passages are currently loaded against their intended mfTier designation. Canon v3.0 MF audit is in progress — passages being revised to validated ranges. Firebase will be updated when revision pass is complete. Passages are pedagogically functional and have been used in over 900 listens in Snorkl format.

EXT — Subtype, Not Umbrella

Architectural Position — Locked
EXT = one canonized long-form listening family. EXT ≠ the universal template for all extended listening content. EXT enforces constraints appropriate to its purpose — single controlling idea, counter-intuitive hinge, no lists, hard-gated length bands, strict recovery logic. These constraints are correct for EXT. They do not generalize to all future long-form content.

Future long-form families may include short lectures, briefings, multi-point discourse, or content with lists and enumeration. These will carry their own corpusType values and metadata fields. Adding a future long-form family requires no refactoring of existing collections — zero impact on EXT documents.

Access Control

Firestore Security Rules — Identity Model
Identity model: document-based — /userIndex pattern (locked for Phase 2A and pilot). Auth custom claims are NOT used. Identity is resolved by reading /userIndex/{request.auth.uid} to get studentId, then reading /students/{studentId} for authorization decisions. Helper functions getStudentId() and getStudent() implement this in rules.

Student lookup: Any authenticated instructor reads any student by B10 ID — all attempts, notes, assignments, scores, feedback. No class membership required.

Passage visibility (students): Student reads passage only if passage.track == student.track OR passage.track == "both". Enforced at database level, not UI level.

Submission rules: Student can CREATE submission only when b10Id in request matches their own. Student cannot UPDATE any submission doc after creation. Score fields and status are write-protected from the client entirely. Only admin SDK (Cloud Function) can update submission documents.

track immutability: student.track write-protected after creation. Admin override requires non-empty trackOverrideNote.

Groups: Private to owning instructor. Other instructors cannot read another's groups.

Notes: Any instructor who looks up a student may write and read notes. Visible to all instructors who look up that student and to admin.
Scoring Pipeline Invariant — Architectural
Raw transcript only reaches the scoring engine — always, without exception.

STT Pass 1 (Deepgram): disfluency-preserved raw transcript + word-level timestamps. This is the SOLE input to the scoring engine on every response.

STT Pass 2 (AssemblyAI): smooth, readable transcript for instructor review only. Triggered conditionally — never by default. Never routed to the scoring engine under any condition.

Score 1 gate: Pass 2 never triggered on Score 1 responses.

EVALUATION_POLICY block prepended to every Claude API call. Canonical version in /config/aiPolicy.js.

Design Notes

Student-centric model
Student is the primary unit. B10 ID is the access key. Any instructor looks up any student they work with. No class membership, no admin roster, no CUI risk.
Identity model
Document-based /userIndex pattern. Auth custom claims not used. Chosen for Phase 2A and pilot: already operational, handles instructor access model cleanly, instant updates on track changes, sufficient performance at pilot scale (50 students).
SLT access
Full library — no restrictions. assignmentType = "slt" is a label only. Supplemental pool is additive. Text-prompt tasks are generatively infinite — no supplemental pool needed for these.
Scaffold system
Two-axis: 6 discourse frames + 7 grammar structures. Default is holistic. Claude receives technical spec. Student receives plain-language cue only. PSU always implicitly evaluated — argument_structure foregrounds it.
claudeFeedback scope
Written to every attempt from Phase 3 launch. Addresses grammar, structure, MF vocab, abstraction, discourse, fluency. Scaffold-aware when scaffoldConfig present. Feedback quality is not deferred. Analytics are.
STT provider
Deepgram Pass 1 (default, every response): disfluency-preserved raw transcript with word-level timestamps. AssemblyAI Pass 2 (conditional only): smooth transcript for instructor review. Pass 2 never routed to scoring engine. Score 1 gate: Pass 2 never triggered on Score 1.
Score ranges
ESO (LEVEL3): integer 2–4 (Score 1 defined but never triggered — Level 2 assumed as floor). Narration/Description/Instructions (LEVEL2): integer 1–3. taskFamily field drives score validation in ClaudeScorer.js and denominator display in student UI.
MF — passage input
Pre-verified externally. Claude does not classify passage vocabulary. mfTier = Canon v3.0 locked ranges on COR (Tier 0: 1–2 · Tier 1: 3–6 · Tier 2: 6–10). ORI: mfTier = "n/a" — no MF system. EXT: mfTier = "n/a" per EXT-SPEC-DELTA v1.3. CORE MF audit in progress — passages being revised to validated ranges.
MF — student output
Phase 4 feature. Claude evaluates student transcript against pre-verified lexicon. Populates mfVocabUsed / mfVocabMissed. Claude as rater of student production — not corpus classifier.
ILR target values
"2" and "3" only. No "2+". 2+ is an OPI rating outcome. mfTier scaffolds within level 3 — all tier 0/1/2 passages are ilrTarget = "3".
EXT subtype
EXT is one canonized long-form family — not the umbrella. Future long-form families carry their own corpus types. Zero refactoring needed to add them.
Future task types
reported_speech_2 (ILR 2 · concrete topic) and reported_speech_3 (ILR 3 · abstract topic) are future taskType values. Not yet in enum — added when corpus is ready. No migration needed.
Instructor autonomy
Platform informs sessions — never prescribes them. No gates, no alerts, no pressure. If an instructor feels boxed in, they will not use it.
Audio TTL
Track A only. Deleted after 2 weeks. Transcript + feedback + features persist permanently.
Mobile-first
Full functionality on iPhone and Android. PWA installable. iOS Safari and Android Chrome tested before Phase 3 launch.
Session note summary
On-demand only — instructor clicks "Summarize notes" to generate. Claude synthesizes all session notes for that student into 3–5 sentences. Result cached in students.studentSummary until instructor clicks Refresh. Never auto-generates on note write.
Privacy / CUI
B10 ID is system-generated — not linked to school ID. Student shares voluntarily. No roster. No sensitive institutional data in the system.
